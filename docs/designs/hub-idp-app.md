## Vision

### 10x Check
The 10x version isn't just an auth gateway — it's the platform brain. It knows every student across every app, tracks their learning journey holistically, enables cross-app intelligence ("use your vocab words in your stories"), and gives parents a single dashboard showing their child's progress across the entire 11+ prep suite. Subscription management, email communications, and admin operations all live here. Adding a new app to the platform = register it in the hub, deploy it, done. Zero auth code in any app, ever.

## Key Decisions (from eng review)
- User schema: adopt vocab-master's (INTEGER IDs, username-based). Email is a required field even though username is the display handle — needed for password reset, notifications, and Google OAuth linking
- DB: each app owns its own DB, hub owns identity (no shared DB)
- Deploy: independent CI/CD per app (GitHub → PR → merge → build → deploy). Hub gets its own GitHub Actions workflow mirroring the existing app pattern
- OIDC: full OIDC provider via `oidc-provider` (panva). Note: this package has 100+ config knobs and requires a custom PostgreSQL adapter for session/token persistence — time-box adapter work or find a reference implementation
- Google OAuth: lives only in hub (federated IdP). Cutover: run parallel (hub + vocab-master both accept Google) during migration, then remove from vocab-master. Existing Google OAuth client IDs must be updated to point at hub's callback URL
- SSO mechanism: apps send OIDC authorize request with `prompt=none`; if hub session cookie (Domain=.labf.app) is valid, token is returned without user interaction; if expired, redirect to hub login page
- Auth client: `@labf/auth-client` SDK published on GitHub Packages under the `@DanWangDev` scope (or `@labf` if org is created)
- Local user sync: each child app stores a local `app_users` table with `hub_user_id` (FK by convention) + app-specific preferences. Identity data (name, email) is read from JWT claims only, never duplicated
- OIDC signing keys: stored as env secret (`OIDC_SIGNING_KEY`). Key rotation strategy deferred — will be needed before multi-instance deployment
- Hub session cookie: `Domain=.labf.app`, HttpOnly, Secure, SameSite=Lax — must be configured from day one
- Subscriptions in Phase 1: admin-assigned via admin panel. No self-service billing until Stripe is added

## Data Model (core entities)

```sql
-- Identity
users (
  id            SERIAL PRIMARY KEY,
  username      TEXT UNIQUE NOT NULL,
  email         TEXT UNIQUE NOT NULL,
  password_hash TEXT,                    -- bcrypt (nullable for Google-only)
  display_name  TEXT NOT NULL,
  role          TEXT NOT NULL DEFAULT 'student',  -- student | parent | admin
  parent_id     INTEGER REFERENCES users(id),
  google_id     TEXT UNIQUE,             -- presence indicates Google-linked account
  email_verified BOOLEAN DEFAULT false,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
)

-- OIDC (managed by oidc-provider adapter)
oidc_sessions, oidc_access_tokens, oidc_authorization_codes,
oidc_refresh_tokens, oidc_device_codes
-- (schema defined by oidc-provider PostgreSQL adapter)

-- NOTE: refresh tokens are managed by oidc-provider's adapter (oidc_refresh_tokens).
-- No separate refresh_tokens table needed.

-- Service-to-service tokens (for app→hub API calls like learning events, stats)
service_tokens (
  id            SERIAL PRIMARY KEY,
  app_id        INTEGER REFERENCES applications(id),
  token_hash    TEXT NOT NULL,           -- SHA256 hash, issued during app registration
  scopes        TEXT[] DEFAULT '{}',     -- 'learning_events:write' | 'stats:read'
  expires_at    TIMESTAMPTZ,            -- NULL = no expiry (rotate manually)
  created_at    TIMESTAMPTZ DEFAULT now()
)

-- Application Registry
applications (
  id              SERIAL PRIMARY KEY,
  name            TEXT NOT NULL,
  slug            TEXT UNIQUE NOT NULL,
  url             TEXT NOT NULL,
  client_id       TEXT UNIQUE NOT NULL,
  client_secret_hash TEXT NOT NULL,      -- bcrypt
  redirect_uris   TEXT[] NOT NULL,
  icon_url        TEXT,
  stats_api_url   TEXT,                  -- e.g., https://vocab-master.labf.app/api/stats
  status          TEXT DEFAULT 'active',
  created_at      TIMESTAMPTZ DEFAULT now()
)

-- Subscriptions
subscriptions (
  id            SERIAL PRIMARY KEY,
  user_id       INTEGER REFERENCES users(id),
  plan          TEXT NOT NULL DEFAULT 'free',  -- free|writing|vocab|bundle|family
  status        TEXT NOT NULL DEFAULT 'active', -- active|trial|expired|cancelled
  features      TEXT[] DEFAULT '{}',     -- computed from plan at token generation, override-able by admin
  expires_at    TIMESTAMPTZ,
  assigned_by   INTEGER REFERENCES users(id),  -- admin who assigned (Phase 1)
  created_at    TIMESTAMPTZ DEFAULT now()
)

-- User ↔ App access (populated automatically from subscription plan;
-- e.g., plan='bundle' → access to all apps. Admin can also grant/revoke manually)
user_app_access (
  user_id       INTEGER REFERENCES users(id),
  app_id        INTEGER REFERENCES applications(id),
  granted_at    TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (user_id, app_id)
)

-- Cross-app Learning Profiles
learning_events (
  id            SERIAL PRIMARY KEY,
  user_id       INTEGER REFERENCES users(id),
  app_id        INTEGER REFERENCES applications(id),
  event_type    TEXT NOT NULL,           -- 'technique_mastered' | 'word_learned' | 'story_completed'
  metadata      JSONB NOT NULL DEFAULT '{}',
  created_at    TIMESTAMPTZ DEFAULT now()
)

-- Audit Log
audit_log (
  id            SERIAL PRIMARY KEY,
  actor_id      INTEGER REFERENCES users(id),
  action        TEXT NOT NULL,           -- 'login' | 'register' | 'impersonate' | 'subscription_change'
  target_id     INTEGER,
  details       JSONB DEFAULT '{}',
  ip_address    TEXT,
  created_at    TIMESTAMPTZ DEFAULT now()
)

-- Password Reset
password_reset_tokens (
  id            SERIAL PRIMARY KEY,
  user_id       INTEGER REFERENCES users(id),
  selector      TEXT UNIQUE NOT NULL,
  validator_hash TEXT NOT NULL,
  expires_at    TIMESTAMPTZ NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT now()
)
```

## SDK Contract (@labf/auth-client)

```typescript
// Express middleware
import { createAuthMiddleware, requireAuth, optionalAuth } from '@labf/auth-client'

// Initialize — call once at app startup
const auth = createAuthMiddleware({
  issuer: 'https://hub.labf.app',       // OIDC issuer URL
  clientId: process.env.OIDC_CLIENT_ID,
  clientSecret: process.env.OIDC_CLIENT_SECRET,
  redirectUri: 'https://my-app.labf.app/auth/callback',
  postLogoutUri: 'https://my-app.labf.app',
})

// Routes
app.get('/auth/callback', auth.callback())   // OIDC callback handler
app.get('/auth/login', auth.login())         // Redirect to hub login
app.get('/auth/logout', auth.logout())       // Logout + redirect to hub
app.use('/api', auth.requireAuth())          // Protect API routes — 401 if no valid session
app.use('/api', auth.optionalAuth())         // Attach user if present, don't block

// Access user in route handlers
app.get('/api/me', auth.requireAuth(), (req, res) => {
  // req.user = { sub, email, username, role, plan, apps, features }
  res.json(req.user)
})

// Frontend helper
import { getLoginUrl, getLogoutUrl, isAuthenticated } from '@labf/auth-client/browser'
```

**SDK internals:** The SDK stores the OIDC access token + refresh token in an encrypted httpOnly session cookie on the app's own domain. On each `requireAuth()` call, it validates the access token locally (JWT signature check using the hub's JWKS). When the 15-min access token expires, the SDK automatically uses the refresh token to get a new one from the hub's `/token` endpoint (transparent to the route handler). If the refresh also fails, it returns 401 and the frontend redirects to hub login. OIDC scopes requested: `openid profile email`. Custom claims (`plan`, `apps`, `features`) are included via oidc-provider's `claims` configuration.

## Learning Events API

Apps push learning events to the hub via REST (fire-and-forget, non-blocking):

```
POST https://hub.labf.app/api/learning-events
Authorization: Bearer <app-level service token>
Content-Type: application/json

{
  "user_id": 42,
  "event_type": "technique_mastered",
  "metadata": {
    "technique": "sensory_detail",
    "source": "coaching_pass_3",
    "confidence": 0.85
  }
}

Response: 201 Created (or 202 Accepted if queued)
```

Apps authenticate with a service-to-service token (separate from user tokens) issued during app registration.

## Parent Dashboard Stats API Contract

Each app exposes a stats endpoint that the hub calls to aggregate progress:

```
GET https://vocab-master.labf.app/api/stats/:hubUserId
Authorization: Bearer <hub-service-token>

Response: {
  "app": "vocab-master",
  "user_id": 42,
  "period": "last_7_days",
  "stats": {
    "words_learned": 47,
    "streak_days": 5,
    "sessions_completed": 12,
    "accuracy": 0.82
  }
}
```

Hub aggregates stats from all registered apps with a `stats_api_url`.

## Migration Strategy

1. **Build hub and deploy** — hub.labf.app goes live with OIDC endpoints + admin panel
2. **Export vocab-master users** — SQL script exports users table to CSV/JSON. Password hashes (bcrypt) are portable — no re-hashing needed
3. **Import into hub PostgreSQL** — Script inserts users, preserving IDs. All users get `plan: 'free'` subscription
4. **Invalidate all refresh tokens** — Force re-login on both apps. Clean cutover, no stale session complexity
5. **Parallel Google OAuth run** — Hub AND vocab-master both accept Google sign-in during migration window. Hub creates user if new; vocab-master continues working. This prevents a broken-Google-login window
6. **Migrate vocab-master** — Install `@labf/auth-client`, remove local auth. Google OAuth client config updated to hub callback URL
7. **Migrate writing-buddy** — Install `@labf/auth-client`, remove local auth. Clean schema reset (no prod data)
8. **Remove Google OAuth from vocab-master** — After confirming hub handles all Google logins
9. **Cleanup** — Delete old auth code from both apps

**Phase C: Expansions (after Phase B, can be parallelized)**

**Phase D: Cross-app Intelligence (after Phase C)**
- Learning events API (hub-side storage + endpoint)
- Child app integration (vocab-master + writing-buddy push events)
- Parent dashboard (aggregates stats from each app's stats API)

## Non-Functional Requirements

- **Auth endpoint latency:** p99 < 500ms for /token, /authorize, /userinfo
- **Session TTL:** Hub session cookie: 7 days. Access tokens: 15 minutes. Refresh tokens: 7 days
- **Password policy:** Minimum 8 characters (matching vocab-master). Bcrypt with 12 rounds
- **Rate limiting:** Login: 5 attempts per 15 minutes per IP. Registration: 3 per hour per IP. Token endpoint: 60 per minute per client_id
- **Uptime target:** 99.5% (hub downtime = no new logins, but existing tokens continue working)
- **Constraint:** All apps must be on `*.labf.app` subdomains for SSO cookie to work. Non-labf.app deployments (preview URLs, white-label) cannot use silent auth

## Reviewer Concerns

3 items from adversarial spec review (non-blocking, address during implementation):

1. **oidc-provider PostgreSQL adapter is non-trivial.** The `panva/node-oidc-provider` package provides a reference MongoDB adapter but not PostgreSQL. Writing a production-ready PG adapter is a sprint-sized deliverable. Mitigation: time-box to 1 week; if too complex, evaluate `oidc-provider`'s in-memory adapter with a Redis persistence layer as fallback, or search for community PG adapters.
2. **Email verification flow is unspecified.** The `email_verified` field exists but the verification trigger, token table, and UX flow are not defined. Resolve during Phase A implementation — standard link-based email verification via Resend.
3. **Learning event loss is acceptable.** The fire-and-forget push from apps to hub means events are lost if the hub is down. This is acceptable at current scale — learning events are supplementary analytics, not source-of-truth data. Add a retry queue if event volume becomes significant.
