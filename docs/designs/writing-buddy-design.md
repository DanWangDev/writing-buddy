## Problem Statement

11+ students need structured, engaging writing practice that goes beyond paper workbooks and occasional tutor sessions. Current tools either offer sterile AI corrections that strip the child's voice, or generic writing prompts with no feedback loop. There's nothing that combines creative writing engagement with iterative, human-feeling AI coaching specifically tuned for the 11+ age group.

This is the second app in a growing 11+ prep suite alongside vocab-master. The apps should share user infrastructure and be architecturally ready for bundling, subscription models, and cross-app features.

## What Makes This Cool

The killer insight: **AI coaching should preserve the child's voice, not replace it.** Most AI writing tools rewrite student work into "perfect" prose — teaching copying, not writing. Writing Buddy takes a different approach: iterative passes where the AI acts like a patient human tutor. It asks questions about the student's choices, suggests improvements in the student's own register, and guides them to revise their own work. Each pass gets warmer and more specific, not more mechanical.

Combined with gamification proven in vocab-master (streaks, XP, achievements, leaderboards), this makes writing practice genuinely engaging for 10-11 year olds.

## Recommended Approach

**Approach B: Shared-Auth Gateway** — the Goldilocks option. It keeps writing-buddy independent enough to develop and deploy without risking vocab-master, while sharing the foundational user infrastructure that makes bundling, subscriptions, and cross-app features natural rather than a retrofit.

### Database Architecture

Both apps connect to a **single shared SQLite database** for user management, auth, and subscription data. Writing-specific tables (prompts, submissions, revisions, coaching_passes) live in the same database alongside vocab-master's tables. This avoids the user-sync problem entirely — there is one `users` table, one `subscriptions` table, one source of truth.

Both Express servers connect to the same SQLite file. To handle concurrent writes safely, the database uses **WAL (Write-Ahead Logging) mode** (`PRAGMA journal_mode=WAL`), which allows concurrent readers with a single writer — sufficient for the expected load (classroom-scale, not internet-scale). Both apps run behind the same nginx reverse proxy (as vocab-master already does), and can optionally run as a single Express process with separate route prefixes (`/api/vocab/...`, `/api/writing/...`) to eliminate SQLite concurrency concerns entirely.

The `shared-auth` package provides the auth middleware, JWT handling, and user/subscription repository interfaces. Both apps import it as a workspace dependency and operate against the shared database.

### Shared-Auth Distribution

`shared-auth` is a **TypeScript package linked via workspace protocol** (e.g., `"shared-auth": "workspace:*"` in package.json). It is NOT a standalone service or microservice. Both vocab-master's backend and writing-buddy's backend import it as a local dependency. It provides: JWT generation/validation, auth middleware, user repository interfaces, subscription checking, Google OAuth, and Turnstile bot protection.

The extraction from vocab-master is non-breaking — the existing auth code moves into the shared package, and vocab-master's backend imports from the new location. However, once both apps depend on `shared-auth`, version bumps to the shared package require coordinated testing and deploys of both apps. **Risk mitigation:** Run vocab-master's full test suite after extraction before merging. The shared-auth package should have its own unit tests covering JWT, middleware, and user operations.

### Cross-App SSO Architecture

All apps live on subdomains of `*.labf.app` (e.g., `vocab-master.labf.app`, `writing-buddy.labf.app`). SSO works via a **shared HttpOnly cookie on the `.labf.app` parent domain** — no hub app, no OAuth redirects between your own apps, no message queues.

**How it works:**

1. **Login (any app):** User signs in on any app → backend generates a JWT access token + refresh token → both set as HttpOnly cookies on `.labf.app` domain.

2. **Cross-app access:** When the user visits another subdomain (e.g., `writing-buddy.labf.app`), the browser automatically sends the JWT cookie. Writing-buddy's backend validates it using the same `shared-auth` middleware and the same signing secret. User is already logged in — zero friction.

3. **JWT claims carry entitlements:**
```typescript
// JWT payload structure
interface JwtPayload {
  sub: string              // user ID
  email: string
  role: 'student' | 'parent' | 'tutor'
  plan: Subscription['plan']  // 'free' | 'writing' | 'vocab' | 'bundle' | 'family'
  apps: string[]           // ['vocab', 'writing'] — which apps this user can access
  features: string[]       // ['flashcards', 'ai-coaching', 'exam-mode'] — granular feature flags
  iat: number
  exp: number              // short-lived: 15 minutes
}
```

4. **Feature gating per app:** Each app's middleware checks `apps` and `features` in the JWT. If a user visits writing-buddy but their `apps` array doesn't include `'writing'`, they see an upsell page, not a login wall. They're authenticated but not entitled — important distinction.

5. **Token refresh:** Refresh tokens (longer-lived, e.g., 7 days) are stored as a separate HttpOnly cookie. Both apps hit the same `/auth/refresh` endpoint (served by whichever backend the request lands on, since both import `shared-auth` and connect to the shared DB). The refreshed JWT includes the latest subscription state — so if a user upgrades mid-session, the new entitlements appear on next refresh without re-login.

6. **Adding future apps:** Deploy a new subdomain on `*.labf.app`, import `shared-auth`, use the same JWT signing secret. Add the new app identifier to the subscription system. Existing apps require zero code changes. The JWT `apps` array is dynamic — populated from the DB at token generation time, not hardcoded.

**Why not a hub app (yet):**
A hub portal (single entry point listing all apps) becomes valuable when there are 3+ apps and users need help navigating the suite. For 2 apps, SSO is sufficient — users bookmark each app directly and are always logged in. The hub can be added as a future phase as simple UI on top of this same SSO foundation, without changing the auth layer.

**Security considerations:**
- JWT signing secret must be identical across all app deployments (store in shared env/secret manager)
- HttpOnly + Secure + SameSite=Lax cookies prevent XSS and CSRF on the shared domain
- Each app still validates the JWT independently — a compromised app can't forge tokens for others without the signing secret
- Refresh token rotation: each refresh invalidates the old refresh token (stored in DB) to limit damage from token theft

### Content Safety (Required for Launch)

For an app targeting 10-11 year olds, content safety is a launch requirement, not an open question.

**Default approach:**
1. **Input screening:** Before sending student writing to the coaching LLM, run it through a lightweight content classifier (LLM-based or rules-based) to flag harmful, violent, or inappropriate content. Flagged content is held for parent/tutor review rather than rejected outright — a child writing about a "battle scene" in a fantasy story is fine, but needs different handling than genuinely concerning content.
2. **Output filtering:** All AI coach responses are filtered for age-appropriateness. The coaching system prompt explicitly constrains vocabulary, tone, and topic boundaries.
3. **Parent visibility:** Parents can view all their child's writing and AI interactions in the parent dashboard.
4. **Audit log:** All coaching interactions are stored (already captured in `coaching_passes` table) for review.
5. **Data retention & deletion (UK Children's Code / GDPR):** Parents can request deletion of all their child's data via the parent dashboard. Deletion removes all submissions, revisions, coaching passes, and progress records for that user. Coaching interaction logs are retained for a maximum of 2 years, then auto-purged. Account deletion is immediate and irreversible. These policies must be documented in the app's privacy policy.

### The AI Coaching Engine — Core Innovation

The coaching engine is the heart of the product. It works as a **conversation between the student and the AI coach**, triggered explicitly by the student.

**How it works (UX flow):**
1. Student picks a prompt and writes in the Writing Desk (minimum 50 words before coaching unlocks)
2. Student clicks "Get Feedback" button → triggers Pass 1 (Warm Acknowledgment)
3. Student reads feedback, optionally revises their writing, clicks "Get More Feedback" → triggers Pass 2
4. Repeat: each "Get Feedback" click triggers the next pass type in sequence
5. After Pass 4 (or when the student clicks "I'm Done"), the submission is marked complete and XP is awarded

**A "coaching session"** = one complete cycle through Passes 1-4 on a single submission. The free tier allows 3 coaching sessions per day, meaning 3 different submissions can go through the full feedback cycle. Each session involves up to 4 LLM calls (one per pass).

**Revision/pass relationship:** Revisions and coaching passes interleave but are NOT 1:1. A student may revise multiple times between requesting feedback, or request feedback without revising. The `coaching_passes.revision_number` field records which version of the student's text the AI was responding to, creating an auditable trail of what the coach "saw" at each step.

The coaching passes are:

**Pass 1 — Warm Acknowledgment:**
Read the student's writing. Identify 2-3 things they did well. Name them specifically ("You used a great metaphor when you compared the fog to a sleeping cat"). No corrections yet.

**Pass 2 — Guiding Questions:**
Ask 2-3 questions that guide the student toward their own improvements. Not "fix this" but "What if your reader could hear the sounds in this scene? What would they hear?" Focus on one dimension at a time (e.g., sensory detail, or character motivation, or story structure).

**Pass 3 — Gentle Suggestions:**
After the student revises based on questions, offer specific but gentle suggestions. "Your revision added great sounds! One more idea: instead of 'he walked slowly,' what word could show us HOW he walked? Was he creeping? Shuffling? Tiptoeing?" Preserve their vocabulary level — don't suggest words a 10-year-old wouldn't use.

**Pass 4+ — Polish & Celebrate:**
Refine remaining rough edges. Celebrate growth: "Look at the difference between your first version and now — you added sensory details, stronger verbs, and your ending gives me chills."

**Key principles:**
- Never rewrite the student's work — only suggest, question, and encourage
- Match the student's vocabulary register (don't suggest "melancholy" to a child who writes "sad")
- Limit feedback to 2-3 points per pass (cognitive overload kills motivation)
- Each pass should feel like a conversation, not a grade report
- Track which techniques the student masters over time (show growth)

### Coaching Prompt Architecture

Each pass uses a structured LLM prompt with this context window strategy:

```
[System prompt — pass-specific instructions + persona]
[Student profile — age, writing level, techniques mastered, vocab level]
[The writing prompt that was given]
[Full revision history — original + all revisions]
[All prior coaching passes for this submission]
[Pass-specific instruction — what to focus on this pass]
```

**System prompt skeleton (Pass 1 — Warm Acknowledgment):**
```
You are a warm, encouraging writing tutor for a {age}-year-old student
preparing for their 11+ exam. Your name is Writing Buddy.

YOUR ROLE: Read the student's creative writing and identify 2-3 specific
things they did well. Name each strength with a concrete quote from their
work. Do NOT suggest any improvements in this pass.

VOICE RULES:
- Write like a friendly teacher, not a textbook
- Use simple vocabulary matching the student's level
- Be specific — "You used a great metaphor when you said X" not "Good use of figurative language"
- Maximum 150 words
- No bullet points or numbered lists — write in natural paragraphs
- End with a question that shows curiosity about their creative choices

DO NOT: rewrite any of their text, use academic jargon, give a grade or score,
mention grammar or spelling (that comes later), suggest improvements.
```

**Session state management:** All revisions and coaching passes are stored in the database (see `revisions` and `coaching_passes` tables). When a student returns to continue a coaching session — even days later — the full context is reconstructed from the DB. To manage token limits:
- Maximum supported writing length: ~2000 words (~3000 tokens)
- **Context scaling strategy by pass:**
  - Pass 1: system prompt (~200) + student profile (~100) + prompt (~100) + original text (~3000) = ~3400 tokens
  - Pass 2: same + Pass 1 feedback (~200) + latest revision (~3000) + original summarized (~300) = ~3900 tokens
  - Pass 3: same + Passes 1-2 feedback (~400) + latest revision (~3000) + prior revisions summarized (~500) = ~4500 tokens
  - Pass 4: same + Passes 1-3 feedback (~600) + latest revision (~3000) + prior revisions summarized (~600) = ~5200 tokens
- **Summarization rule:** Only the latest revision is sent in full. Prior revisions are summarized to a ~200 token diff summary ("Changes from v1→v2: added sensory details in paragraph 2, changed ending")
- **Per-call token budget:** ~6000 tokens input + ~300 tokens output (hard limit enforced server-side)
- **Per-session cost (4 passes):** ~25,000 total tokens × ~$0.25/1M (Haiku input) ≈ $0.006 input + ~$1.25/1M × 1200 output tokens ≈ $0.002 = **~$0.008 per full session**
- **AI failure handling:** If the LLM API is unavailable or returns an error, the student sees "Your writing coach is taking a break — try again in a few minutes." Writing and revision continue to work; only coaching is blocked. If output filtering catches inappropriate content, the pass is silently regenerated (max 2 retries, then generic fallback: "Great work so far! Save your progress and let's continue next time.")

### AI Cost Controls

- **Free tier:** 3 coaching sessions per day (a "session" = one full pass cycle on a submission)
- **Paid tier:** Unlimited coaching sessions
- **Per-call token budget:** ~6300 tokens (hard limit enforced server-side)
- **Daily spend ceiling:** Configurable per-user and global. Default: $0.10/user/day (≈12 full sessions), $50/day global
- **Circuit breaker:** If global daily spend exceeds ceiling, new coaching requests return a friendly "Coach is resting" message. No queuing — students can continue writing and request coaching when the ceiling resets at midnight UTC
- **Model selection:** Default to cost-efficient model (Claude Haiku / GPT-4o-mini). Premium model available as paid feature

### Subscription Architecture (Day 1 Ready)

```typescript
// shared-types/subscription.ts
interface Subscription {
  id: string
  userId: string
  plan: 'free' | 'writing' | 'vocab' | 'bundle' | 'family'
  status: 'active' | 'trial' | 'expired' | 'cancelled'
  features: string[]  // granular feature flags
  expiresAt: Date
  billingProvider?: 'stripe' | 'apple' | 'google'
}

// Feature gating — flexible for any future pricing model
interface FeatureGate {
  feature: string           // e.g., 'ai-coaching', 'exam-mode', 'peer-review'
  requiredPlan: Array<Subscription['plan']>  // plans that include this feature
  trialAllowed: boolean     // available during free trial?
  limit?: number            // e.g., 3 AI coaching sessions/day on free tier
}
```

Subscription **types** live in `shared-types/` (imported by both frontends and backends). Subscription **validation logic and feature-gating middleware** live in `shared-auth/` (imported by backends only). Both apps check entitlements against the same user record in the shared database. Free tier gets limited AI coaching sessions per day; paid tier gets unlimited + exam mode + social features.
